# MISCOVER — Build Spec (V1.0)

You are building Miscover. A taste decoder. Three inputs, one output. One page. One purpose.

This document is the single source of truth for building Miscover's MVP. It combines product vision, technical spec, and acceptance criteria into one file. A developer who has never spoken to anyone on this project should be able to build the entire MVP from this document alone, with zero clarification needed.

If something is ambiguous, it's a bug in this document. Flag it.

---

## What We're Building

A single-page web app where a user types three things they like, presses GO, and receives a three-part output (decode, world, brief) generated by the Claude API. No accounts. No navigation. No explanation. The output is designed to be screenshotted and shared.

**Core loop:** Type three things → Read decode → Screenshot → Share → "What are your three?"

---

## Mira

Mira Sato is the user every decision is tested against. Read `/mnt/project/mira.md` for her full profile.

### What to extract from Mira's profile:

- **Trust moments:** The first sentence of the decode is what earns or loses her. If it's generic, she closes the tab. If it's specific, she screenshots. The decode has roughly 3 seconds to prove it knows her.
- **Failure modes:** She abandons if the site explains itself, asks for signup, shows a loading state longer than ~4 seconds, or produces output that "could describe anyone." She is also lost forever if the World section recommends something obvious — if she typed Wong Kar-wai and the World suggests In the Mood for Love, the product has failed.
- **Four test input sets** (use these for QA):
  - Set 1: `Wong Kar-wai, Aesop, concrete`
  - Set 2: `Tirzah, onsen, Dieter Rams`
  - Set 3: `Dean Blunt, convenience stores, Helmut Lang`
  - Set 4: `Bristol, 35mm, miso soup`
- **Validation rule:** The decodes for all four sets must feel like four facets of the same person. If they don't, the system prompt needs tuning, not the frontend.

### The Mira Test

Before building, writing, or suggesting anything:

> *Would Mira screenshot this and post it to her Instagram Story with no caption — just the screenshot — and feel like it says something about her?*

If no, it is not ready.

## Page Structure

The page has exactly two states: **input** and **result**.

### Input state

```
[                              ]
        ___________________
       |   first thing     |
       |___________________|
        ___________________
       |   second thing    |
       |___________________|
        ___________________
       |   third thing     |
       |___________________|

            [  GO  ]
```

**Specifications:**

| Element | Spec |
|---|---|
| Background | `#111` solid. No gradient, no texture. |
| Page layout | Flexbox, centered vertically and horizontally. `min-height: 100vh`. |
| Input fields | `width: 280px`. Bottom border only: `1.5px solid #333`. No background. No outline on focus — border changes to `#888`. |
| Input font | Courier Prime, `17px`, `#ccc`, `text-align: center`, `letter-spacing: 0.02em`. |
| Placeholder text | "first thing" / "second thing" / "third thing". Color `#444`. Not italic. |
| Input spacing | `16px` gap between fields. |
| GO button | `padding: 10px 48px`. Background `#ccc`, text `#111`. Courier Prime `14px`, `letter-spacing: 0.2em`. |
| GO disabled state | Background `#333`. No cursor pointer. Enabled only when all three fields have non-empty trimmed values. |
| GO hover | Background `#fff`. |
| Tab/Enter behavior | Enter or Tab on field 1 → focus field 2 (if field 1 non-empty). Enter or Tab on field 2 → focus field 3 (if field 2 non-empty). Enter on field 3 → submit (if all fields non-empty). Shift+Tab navigates backward. |
| Autofocus | Field 1 on page load. |

### Loading state

Replaces input state entirely. Centered on page:

```
          ...
```

Three dots, Courier Prime `18px`, color `#555`, each with a pulse animation (opacity 0.2 → 1 → 0.2) staggered by 0.2s. `animation: pulse 1.4s infinite ease-in-out`.

Replaces loading state. Centered on page, `max-width: 520px`:

        Brand — Aesop
        Font — Founders Grotesk
        Album — Dummy
        City — Kyoto
        Photographer — Rinko Kawauchi

        dense, specific, functional.

**Specifications:**

| Element | Spec |
|---|---|
| Container | `max-width: 520px`, `width: 100%`. Fade-up animation: `opacity 0 → 1`, `translateY(20px) → 0`, `0.8s ease`. |
| Decode text | Courier Prime `16px`, `line-height: 1.7`, color `#ccc`, `text-align: center`. |
| Separator | `div`, `width: 40px`, `height: 1px`, background `#333`, `margin: 32px auto`. |

## The System Prompt

This is the most important piece of engineering in the product. The quality of the decode depends entirely on this prompt. It must be tuned obsessively.

### Current prompt (V1.0):

```
you are the reader. you observe three things someone chose and find the thread that connects them. you are not enthusiastic. you are not mean. you are flat, certain, and precise. you state observations the way a doctor reads test results. no hedging. no "perhaps." no "it seems like." you just say it.

your response has three sections, separated by ---

SECTION 1: THE DECODE
2-3 sentences. lowercase. no exclamation marks. start with the connection, not the inputs. never begin with "your three choices" or "these three things." never name all three inputs back. the user knows what they typed. use at least one single-word sentence. second person present tense. "you want" not "this suggests." no compliments. no "great choices." the first sentence should be something the user has never articulated about themselves but immediately recognizes as true.

never use these words: fascinating, reveals, unveils, journey, unique, curated, resonates, speaks to, energy, aesthetic, vibe. never use "at the intersection of." never hedge with "might" or "could be."

SECTION 2: YOUR WORLD (exactly 8 items)
8 references across different domains. format: "Domain — Name" (e.g., "Film — Heat"). no explanations. no parentheticals. at least 5 different domains. no more than 2 from the same domain. avoid the obvious — find the adjacent thing they haven't discovered yet. pick from: Film, Director, Music, Album, Artist, Architect, Building, Brand, Font, City, Neighborhood, Restaurant, Hotel, Book, Author, Photographer, Designer, Magazine, Color, Material, Decade, Texture, Fragrance, Car, Game.

SECTION 3: YOUR BRIEF
one paragraph. lowercase. 4-6 sentences. dense with specific imagery — "brushed steel, not chrome" not "high quality materials." written so someone could paste it into midjourney, chatgpt, a design brief, or a figma file and get the right output. no generic descriptors. every phrase should narrow the field.

no headers, no labels, no bullet points. just the three sections separated by ---.
```

### Prompt tuning criteria:

Run Mira's four test sets through the prompt. Evaluate each decode against:

1. **Specificity:** Could this decode describe someone who typed three different things? If yes, it's too generic. Fail.
2. **Recognition:** Does the first sentence state something that feels true but unspoken? If it reads like a horoscope, fail.
3. **Differentiation:** Do Sets 1-4 produce meaningfully different decodes? If any two feel interchangeable, the prompt isn't sensitive enough to the inputs. Fail.
4. **Cohesion:** Do all four decodes feel like facets of one person? If they feel like four strangers, the prompt is too reactive to individual inputs and not finding deep enough through-lines. Fail.
5. **Banned words:** Does the output contain any word from the banned list? Automatic fail.
6. **Structural compliance:** Lowercase? Single-word sentence present? Three sections separated by `---`? Eight world items? Brief is one paragraph? Check all.

## API Integration

### Request

```javascript
const response = await fetch("https://api.anthropic.com/v1/messages", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({
    model: "claude-sonnet-4-20250514",
    max_tokens: 1000,
    system: SYSTEM_PROMPT,
    messages: [
      {
        role: "user",
        content: `${inputs[0].trim()}, ${inputs[1].trim()}, ${inputs[2].trim()}`
      }
    ]
  })
});
```

### Response parsing

The response body is `data.content`, an array of content blocks. Extract text:

```javascript
const text = data.content
  ?.map((b) => (b.type === "text" ? b.text : ""))
  .join("");
```

Split on `---` separator (with flexible regex for varying whitespace/dashes):

```javascript
const sections = text.split(/\n---\n|\n-{3,}\n/);
const result = {
  decode: (sections[0] || "").trim(),
  world: (sections[1] || "").trim().split("\n").filter((l) => l.trim()),
  brief: (sections[2] || "").trim()
};
```

### Edge cases and failure modes

| Scenario | Handling |
|---|---|
| API returns HTTP error (4xx, 5xx) | Show "nothing came back. try again." in result area. Same font, same color, same position. No error codes, no technical language. Transition back to input state after 3 seconds OR show AGAIN button. |
| API returns 200 but `data.content` is empty or malformed | Same as above: "nothing came back. try again." |
| Response doesn't contain `---` separators (model didn't follow format) | Display entire response as the decode. Leave World and Brief empty. Still functional, just less structured. |
| Response contains only 1 separator (2 sections instead of 3) | Parse what exists. Section 1 = decode, Section 2 = world OR brief (infer from content — if it contains `—` formatted lines, it's world). Missing section is simply not rendered. |
| Network timeout (>10 seconds) | Abort fetch. Show "took too long. try again." |
| User submits while previous request is in flight | Ignore the second submission. The GO button / Enter key should be disabled during loading state. |
| User enters same text in all three fields | Allow it. The API will handle it. Some people will type "red, red, red" out of curiosity. That's fine. |
| User enters very long text (>200 chars per field) | Allow it client-side. The API will handle it. No truncation. |
| User enters numbers, URLs, emoji, non-English text | Allow all. The API will handle it. No input validation beyond "non-empty after trim." |
| User enters potentially offensive inputs | Allow it. The model's built-in safety will handle the output. No client-side content filtering. |
| Clipboard API fails (brief copy) | Fail silently. No toast, no error. The click just does nothing. |

## Performance Targets

| Metric | Target | How to measure |
|---|---|---|
| Time to interactive (page load → first field focusable) | < 1.5 seconds on 3G | Lighthouse / WebPageTest throttled |
| Time to decode (GO press → decode visible) | < 4 seconds median | Measured from `fetch()` call to render. Dependent on API latency. |
| Font load | < 500ms on 4G | Google Fonts with `display=swap`. Courier New as fallback renders instantly. |
| Total page weight | < 100KB (excluding font) | Single component, no framework beyond React. |
| Largest Contentful Paint | < 2 seconds | The page is mostly empty. This should be trivial. |
| Cumulative Layout Shift | 0 | No elements shift after load. The input/result states occupy the same centered position. |

## User Stories (MVP)

### Story 1: Type three things

**As** Mira **I want** to type three things I like into three fields **so that** I can decode my taste.

**Acceptance criteria:**
- [ ] Page loads with three empty text fields, vertically stacked, centered
- [ ] First field is autofocused on page load
- [ ] Placeholder text shows "first thing" / "second thing" / "third thing"
- [ ] Enter on a non-empty field advances focus to the next field
- [ ] Enter on the third non-empty field submits the form
- [ ] Tab on a non-empty field advances focus to the next field
- [ ] GO button is disabled (visually: `#333` background) until all three fields contain non-empty trimmed values
- [ ] GO button is enabled (visually: `#ccc` background) when all three fields are non-empty
- [ ] Typing in any field immediately re-evaluates GO button state

**Edge cases:**
- [ ] Whitespace-only input: `"   "` → treated as empty (GO stays disabled)
- [ ] Paste into field: works normally, triggers GO re-evaluation
- [ ] Enter on empty field: nothing happens (no focus advance, no submit)
- [ ] Tab on empty field: browser default (moves to next focusable element, which is the next field)
- [ ] Very long input (500+ chars): allowed, no truncation, field scrolls horizontally
- [ ] Emoji input: allowed
- [ ] Non-Latin characters: allowed
- [ ] Three identical inputs: allowed

**Performance:**
- [ ] Time from page load to first field focusable: < 1.5s on throttled 3G

**Not included:**
- Input validation beyond empty check
- Character limit
- Autocomplete or suggestions
- Input history

---

### Story 2: See the decode

**As** Mira **I want** to see a decode of my three things **so that** I can understand what they say about me.

**Acceptance criteria:**
- [ ] Pressing GO (or Enter on third field) transitions to loading state
- [ ] Loading state replaces input state entirely (inputs are not visible during loading)
- [ ] Loading indicator: three dots with staggered pulse animation
- [ ] API call fires with the three trimmed inputs joined by commas
- [ ] On successful response, loading state transitions to result state with fade-up animation (0.8s ease)
- [ ] Result state displays decode, separator, world (8 items), separator, brief
- [ ] Decode: Courier Prime 16px, #ccc, centered
- [ ] World items: Courier Prime 14px, #777, centered, one per line
- [ ] Brief: Courier Prime 13px, #777, centered
- [ ] AGAIN button below brief

**Edge cases:**
- [ ] API HTTP error → show "nothing came back. try again." in decode position + AGAIN button
- [ ] API timeout (>10s) → abort fetch, show "took too long. try again." + AGAIN button
- [ ] Malformed response (no `---` separators) → display entire response as decode, skip world/brief sections
- [ ] Partial response (only 1 separator) → parse into 2 sections, infer which is which, render what exists
- [ ] Empty response body → "nothing came back. try again."
- [ ] Response with banned words from lexicon → display as-is (this is a prompt tuning issue, not a frontend issue)
- [ ] Rapid double-submit (user mashes GO) → second submission ignored; GO/Enter disabled during loading

**Performance:**
- [ ] GO press to decode visible: < 4 seconds (median), < 6 seconds (p95)
- [ ] Fade-up animation: 0.8s ease, no jank

**Not included:**
- Streaming / progressive rendering of the decode
- Retry button (AGAIN resets to input; user resubmits manually)
- Caching of results

### Story 3: Copy the brief

**As** Mira **I want** to copy the brief to my clipboard **so that** I can paste it into another tool.

**Acceptance criteria:**
- [ ] Clicking the brief text copies its content to clipboard
- [ ] Brief has `cursor: pointer` to indicate clickability
- [ ] Brief text color lightens on hover (#777 → #999, 0.2s transition)
- [ ] Title attribute on brief element: "Click to copy" (or similar — Mira shouldn't need it, but accessibility)

**Edge cases:**
- [ ] Clipboard API not available (older browser, insecure context) → click does nothing, no error shown
- [ ] Clipboard write fails → silent failure, no error shown
- [ ] User selects text within brief (standard text selection) → normal browser behavior, does not trigger copy of full brief
- [ ] Brief is empty (malformed API response) → click does nothing

**Not included:**
- "Copied!" toast or confirmation
- Copy button icon
- Copy the decode or world sections

### Story 4: Start over

**As** Mira **I want** to try three different things **so that** I can see a different version of myself.

**Acceptance criteria:**
- [ ] AGAIN button visible in result state below the brief
- [ ] Clicking AGAIN transitions to input state
- [ ] All three fields are cleared
- [ ] First field is autofocused after 100ms delay
- [ ] Previous result is cleared from state (not visible if user scrolls)
- [ ] No animation on transition back to input (instant)

**Edge cases:**
- [ ] AGAIN during API call (shouldn't be possible — AGAIN only shows in result state)
- [ ] Rapid AGAIN → GO → AGAIN → GO: each cycle works correctly with clean state
- [ ] Browser back button after AGAIN: no-op (single page, no routing)

**Not included:**
- History of previous decodes
- Undo (going back to previous result)
- Pre-filling fields with previous inputs

---

### Story 5: Screenshot the decode

This is not a feature. It's a quality bar. The result state must be screenshot-worthy by default.

**Acceptance criteria:**
- [ ] Result state on mobile (375px width) fits within one screen without scrolling for typical decode lengths
- [ ] Dark background (#111) extends full viewport — no white bars, no body background bleed
- [ ] No UI chrome in the screenshot zone: the decode, world, and brief are visually self-contained
- [ ] Three inputs are NOT currently shown in the result state. **Decision needed:** Should they be? For screenshot context, the inputs should be visible above the decode. Add them as a single line: `wong kar-wai / aesop / concrete` in #555, Courier Prime 13px, centered, above the decode with 24px margin below.
- [ ] AGAIN button is visually quiet enough that it doesn't degrade the screenshot if included

**Edge cases:**
- [ ] Very long decode (model generates 4+ sentences) → text wraps naturally, page scrolls if needed. Not ideal but acceptable.
- [ ] Very long world item (long name + long domain) → wraps naturally. Monospace handles this.
- [ ] Very long brief → wraps naturally. No truncation.

**Not included:**
- Built-in screenshot tool
- Share sheet
- "Share to Stories" integration
- Watermark or branding in the screenshot zone

## Cross-Reference Conventions

When stories depend on each other or on shared state, use these conventions:

- **State references:** `inputs`, `loading`, `result`, `phase` are the four state variables. All stories reference these.
- **Schema:** Result object shape is `{ decode: string, world: string[], brief: string }`. If this changes, all stories referencing result display must be updated.
- **System prompt changes:** Any change to the system prompt potentially affects Stories 2 and 5 (decode quality and screenshot quality). Re-run Mira's four test sets after every prompt change.
- **Performance changes:** If the API model changes (e.g., from Sonnet to Haiku for speed), Story 2 performance targets apply. Re-validate decode quality with Mira's test sets.

## What We Ship

Miscover ships when:

1. Mira can type three things and read a decode in under 30 seconds total
2. Mira's four test input sets produce four decodes that feel like four facets of one person
3. The decode contains zero banned words from the lexicon
4. The screenshot looks right on a phone screen posted to an Instagram Story with no caption
5. The World section contains at least 3 references per decode that are non-obvious
6. The brief is functional enough to paste into ChatGPT and get outputs that feel right
7. The page loads and is interactive in under 1.5 seconds on 3G
8. Every edge case listed in this document has been tested and handles gracefully

Everything else is iteration.
